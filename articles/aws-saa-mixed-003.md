# インフラ初心者が今週踏み抜いたAWSの地雷まとめ

今週、AWS SAAの設計シナリオに挑戦する中で、実務でも致命傷になりかねない重要な仕様の勘違い（地雷）を踏み抜きました。エンジニアとしての「振り返り」を兼ねて、アーキテクチャの解説と共に資産としてここに記録します。

---

## 🚨 地雷ログ：金融トランザクションの順序保証と重複排除

### 1. 遭遇したシナリオと私の誤認
- **要件:** 銀行振込を非同期で処理するシステム。コンプライアンス要件により「送信された順番通りの厳密な処理」と「重複処理の絶対禁止」が求められる。
- **私のミス:** 「非同期のストリーム処理」という言葉に引きずられ、順序や重複排除の厳密な保証がない**Amazon Kinesis Data Firehose**を選択してしまいました。

### 2. クラウド設計における正解と選定理由
- **最適解:** **Amazon SQS FIFO Queue**
- **アーキテク放談:** 金融系や決済系のシステムにおいて、メッセージの順序入れ替わりや二重処理は致命的な障害（二重送金など）を引き起こします。標準のSQSやKinesisでは「少なくとも1回の配信（Exactly-onceは非保証）」や「ベストエフォートの順序付け」となるため、要件を満たせません。ビジネス要件に「厳密な順序（Exact order）」と「重複排除（No duplicates）」が明記された場合、アーキテクチャの選択肢はメッセージグループIDと重複排除IDを備えた SQS FIFO キュー一択となります。

## 🚨 地雷ログ：一時的なセキュアコンテンツ配信と権限付与の境界

### 1. 遭遇したシナリオと私の誤認
- **要件:** 有料会員に対して、プライベートなS3バケット内の動画や電子書籍を配信する。ダウンロードリンクは15〜30分で自動的に無効化されなければならない。
- **私のミス:** S3バケット自体を一時的に公開したり、一般ユーザーに対して**IAM Role**を付与して直接アクセスさせようとしてしまいました。（※このシナリオで2回連続で失点しました）

### 2. クラウド設計における正解と選定理由
- **最適解:** **Amazon S3 Presigned URL (署名付きURL)**
- **アーキテク放談:** 一般のインターネットユーザー（Webアプリの顧客）にAWS内部の権限システムであるIAM Roleを割り当てるのは、セキュリティと運用の両面で完全なアンチパターンです。また、一時的とはいえS3バケットをパブリックにするのは情報漏洩の温床となります。バックエンドアプリケーション（自身はIAM権限を持つ）が、暗号化された期限付きの「署名付きURL」を動的に生成し、クライアントに渡す。これにより、S3はプライベートのまま、セキュアかつ一時的なアクセス委譲（Delegation）が実現します。

## 🚨 地雷ログ：極限のI/O性能を求める一時キャッシュの罠

### 1. 遭遇したシナリオと私の誤認
- **要件:** リスク分析のシミュレーションにおいて、極めて高いIOPSと最小のレイテンシを誇る一時的なキャッシュスペース（Scratch space）が必要。データはインスタンス停止時に破棄されてよい。
- **私のミス:** 「最高のIOPS」というキーワードに反射し、最も高価なネットワークストレージである**Amazon EBS Provisioned IOPS SSD (io2)**を選択してしまいました。

### 2. クラウド設計における正解と選定理由
- **最適解:** **Amazon EC2 Instance Store**
- **アーキテク放談:** どんなに高性能なEBS（io2）を用意しても、EBSは「ネットワーク越し」にアタッチされる仮想ドライブであるため、物理的なネットワークレイテンシの壁を超えることはできません。一方、Instance StoreはEC2が稼働する物理ホストサーバーに「直接内蔵された」NVMe SSDです。インスタンスを停止するとデータが消滅する（揮発性）という特性がありますが、ビジネス要件が「一時的なデータ（Temporary / Scratch）」であるなら、追加コストゼロで物理限界のI/O性能を引き出せるInstance Storeが最強の選択肢となります。
